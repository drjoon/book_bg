## 실패 원인별 분석

### 1. 이형필 — Pre-login 자체 실패

- 총 8번 시도, 전부 10000ms timeout
- 마지막 deadline(08:59:55)도 초과 → fallback 시도 → `connect ETIMEDOUT 61.76.72.26:443`
- **원인**: 로그인 서버 TCP 연결 불가. 다른 8명은 attempt 1에 전부 성공 (~1.5~3.5s)한 것과 대조적
- Invoke 시각(08:58:39)이 다른 성공자들보다 약 8~11초 늦었고, 해당 Lambda 리전의 네트워크 문제로 보임

### 2. 오수양 · 권택현 · 오대출 — Precision wait 정상, 슬롯 범위 소진

**Precision wait 완료 시각은 성공자들과 동일 수준:**
- 오수양 +409ms, 권택현 +393ms, 오대출 +419ms (성공자 범위: +256~+538ms)
- **타이밍 문제가 아님**

**각 실패자의 슬롯 소진 과정:**

| 이름 | 타겟 범위 | inRange(초기) | 시도 슬롯 | 결과 |
|------|-----------|--------------|-----------|------|
| 오수양 | 0750–1000 | 4 | 0820_A(422), 0752_A(422) | 범위 내 잔여 없음 |
| 권택현 | 1000–1220 | 6 | 1215_B(422), 1147_A(422), 1215_A(422) | 범위 내 잔여 없음 |
| 오대출 | 1110–1300 | 14 | 1229_A·B, 1250_B, 1257_B, 1243_B (전부 422) | 범위 내 잔여 없음 |

모든 422는 **외부 경쟁자(다른 예약자)가 먼저 선점**한 것. 우리 Lambda 간 슬롯 충돌은 DynamoDB claim으로 잘 차단됐으나, 서버 측에서 이미 타인이 예약 완료한 슬롯들이었음.

## 핵심 인사이트

1. **Pre-login 타이밍과 성패는 무관** — 오수양(+1.6s), 권택현(+3.4s) 모두 pre-login 속도가 달랐지만 동일하게 실패. 정광호(+1.5s)와 손덕일(+1.5s)은 가장 늦게 invoke됐음에도 성공.

2. **Precision wait 정밀도는 충분** — 성공/실패 모두 09:00:00 기준 +256ms~+538ms 이내에 슬롯 fetch 시작. CREATE 첫 요청도 09:00:01.2~1.7초 사이로 유사.

3. **실패의 진짜 원인은 `PRIMARY_SLOT_OFFSET` 기반 타겟 범위** — 오수양(0750–1000), 권택현(1000–1220), 오대출(1110–1300)은 해당 시간대 슬롯이 외부 경쟁이 심한 인기 구간이었거나, inRange 슬롯 수 자체가 적었음. 특히 오수양의 inRange가 4개뿐으로 가장 취약.

4. **이형필은 별개 문제** — 네트워크(ETIMEDOUT)로 pre-login 윈도우(08:59:55) 전까지 단 한 번도 로그인 서버 접속을 못 함. Lambda 리전 또는 IP가 일시적으로 차단됐을 가능성.

## 결론

- **타이밍 최적화는 더 이상 개선 여지가 적음** — 모든 성공/실패 케이스가 09:00:00 기준 ~300ms 안팎에서 슬롯 fetch 시작
- **슬롯 선택 전략 개선 필요** — PRIMARY_SLOT_OFFSET 대신 inRange 슬롯 수, 과거 성공률 기반으로 초기 타겟 범위를 동적으로 조정
- **이형필 케이스 대응** — 리전 분산 또는 재시도 로직 고려 (현재 8번 retry 후 fallback, fallback도 실패하면 종료)
